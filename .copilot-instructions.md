# GenomeHubs Data Flows — Coding Conventions

This document outlines the established coding patterns and conventions for the GenomeHubs data pipeline (`flows/`). All new code should adhere to these guidelines to maintain consistency and readability across the codebase.

## **1. Project Structure**

The `flows/` directory organizes code into distinct categories:

```
flows/
├── lib/              # Shared utilities and flow infrastructure
├── parsers/          # Data parsing flows (plugin-based)
├── updaters/         # Data fetching/updating flows
├── validators/       # Data validation flows (emerging pattern)
├── feature_parsers/  # Feature-specific parsing flows
├── orchestration/    # Flow composition and coordination
└── prefect.yaml      # Deployment configuration
```

### **Key Subdirectories**

- **`lib/`**: Core infrastructure
  - `utils.py` — Shared utility functions (Config class, safe_get, parse_s3_file, etc.)
  - `shared_args.py` — Centralized CLI argument definitions
  - `conditional_import.py` — Handles Prefect/non-Prefect execution modes
  - `shared_tasks.py` — Reusable task definitions

- **`parsers/`**: Transform raw data into GenomeHubs-compatible format
  - Plugin-based architecture via `register.py`
  - Standard args: INPUT_PATH, YAML_PATH, APPEND, DATA_FREEZE_PATH
  - All inherit from `flows/parsers/args.py`

- **`updaters/`**: Fetch remote data and emit completion events
  - Standard args: ROOT_TAXID, OUTPUT_PATH, S3_PATH, DATA_FREEZE_PATH, TAXDUMP_PATH
  - Emit events for downstream flow triggering

- **`validators/`**: Validate parsed/processed data against schema (emerging)
  - Standard args: YAML_PATH, TAXDUMP_PATH, MIN_VALID, MIN_ASSIGNED, S3_PATH
  - Current single-file pattern in `lib/validate_file_pair.py`
  - **Future**: Move to `flows/validators/` to match parser/updater pattern

## **2. Imports and Module Resolution**

### **Absolute Package Imports (Required)**

All internal module imports must use absolute package paths to support both:

- Running flows with `python -m flows.lib.<flow>`
- Prefect deployments (which import flows as packages)

✅ **Correct**:

```python
from flows.lib import utils
from flows.lib.conditional_import import flow, task
from flows.lib.shared_args import required, parse_args
from flows.lib.shared_tasks import get_filenames
```

❌ **Incorrect**:

```python
import utils  # Will fail with python -m
from conditional_import import flow  # Ambiguous resolution
```

### **Import Order**

1. Standard library
2. Third-party packages (boto3, requests, etc.)
3. External packages from genomehubs
4. Internal flows package imports

```python
import os
import json
from typing import Optional

import boto3
import requests
from genomehubs import utils as gh_utils

from flows.lib.conditional_import import flow, task
from flows.lib.shared_args import parse_args, required
from flows.lib.utils import Config, safe_get
```

### **Conditional Imports for Prefect Compatibility**

Always import Prefect components via `conditional_import` to allow execution with `SKIP_PREFECT=true`:

```python
from flows.lib.conditional_import import flow, task, emit_event
```

This module provides dummy decorators when `SKIP_PREFECT=true`, enabling local testing without a Prefect server.

## **3. Docstrings**

### **Style: Google Format**

All public functions, classes, and tasks must have docstrings in Google format:

```python
def parse_assembly_report(jsonl_path: str) -> Generator[dict, None, None]:
    """
    Parses an NCBI datasets JSONL file and yields each assembly report.

    Minimal side effects: reads from file, deserializes JSON, yields dicts.

    Args:
        jsonl_path (str): Path to the JSONL file.

    Yields:
        dict: The assembly report data as a dictionary.

    Raises:
        RuntimeError: If the JSONL file cannot be read.
    """
    try:
        with open(jsonl_path, "r") as f:
            for line in f:
                yield utils.convert_keys_to_camel_case(json.loads(line))
    except Exception as e:
        raise RuntimeError(f"Error reading JSONL file: {e}") from e
```

### **Docstring Sections**

- **Summary line**: One-sentence description of purpose
- **Extended description** (optional): Additional context on behavior or side effects
- **Args**: Parameters with types and descriptions
- **Returns** (or **Yields**): Return type and description
- **Raises**: Exception types and when they're raised

### **Requirements**

- ✓ All public functions must have docstrings
- ✓ All Task-decorated functions must have docstrings
- ✓ Include type hints in function signature
- ✓ Describe side effects if any (e.g., "Writes to S3")

## **4. Function Design — Clean, Focused, Minimal Nesting**

### **Single Responsibility Principle**

Each function should have one clear purpose. Use task decomposition for complex workflows:

✅ **Good — Focused**:

```python
@task()
def parse_assembly_report(report: dict, config: Config) -> dict:
    """Extract and transform assembly-level information."""
    # Single concern: parse one report
    return {**report, "processedAssemblyInfo": {...}}

@task()
def validate_report(report: dict) -> bool:
    """Check if report is valid and not atypical."""
    return "assemblyInfo" in report and not report["assemblyInfo"].get("atypical")

@flow()
def process_assemblies(yaml_path: str, work_dir: str):
    """Orchestrate assembly processing pipeline."""
    config = load_config(yaml_path)
    for report in parse_reports():
        if validate_report(report):
            write_output(parse_assembly_report(report, config))
```

❌ **Poor — Too Much Nesting**:

```python
@flow()
def process_assemblies(yaml_path: str, work_dir: str):
    config = load_config(yaml_path)
    for report in parse_reports():
        if "assemblyInfo" in report:
            if not report["assemblyInfo"].get("atypical"):
                processed = {**report, "processedAssemblyInfo": {...}}
                for item in processed.items():
                    if validate_item(item):
                        # More nesting...
```

### **Nesting Limits**

- Maximum 2–3 levels of nesting in control flow (if/for/while)
- Extract inner logic into separate functions beyond that
- Use generators to avoid nested loops where possible

### **Minimize Side Effects**

- Utility functions in `flows/lib/utils.py` should be **read-only** or **write-once**
- Avoid mutations; use `{**dict}` unpacking instead of in-place modifications
- Pass data through the flow rather than storing in module state

✅ **Good**:

```python
def process_record(record: dict) -> dict:
    """Return new dict with processed fields."""
    return {**record, "processed": True, "timestamp": datetime.now()}
```

❌ **Poor**:

```python
def process_record(record: dict) -> None:
    """Mutate record in place."""
    record["processed"] = True
    record["timestamp"] = datetime.now()
```

## **5. Shared Arguments — Dict-Based Convention**

All CLI arguments are defined centrally in `flows/lib/shared_args.py` as dictionaries:

```python
S3_PATH = {
    "flags": ["-s", "--s3_path"],
    "keys": {
        "help": "Path to remote file/directory on S3.",
        "type": str,
    },
}
```

### **Using Shared Args**

In subdirectories, create an `args.py` file that composes shared args:

**`flows/parsers/args.py`**:

```python
from flows.lib.shared_args import INPUT_PATH, YAML_PATH, APPEND, DATA_FREEZE_PATH, parse_args, required

def parse_args(description: str = "A data parser") -> argparse.Namespace:
    """Parse command-line arguments for parsers."""
    return _parse_args(
        [required(INPUT_PATH), required(YAML_PATH), APPEND, DATA_FREEZE_PATH],
        description,
    )
```

### **Argument Modifiers**

Use these helpers to customize shared arguments:

- `required(arg)` — Mark argument as required
- `default(arg, value)` — Set default value and update help text
- `multi(arg)` — Accept multiple values with `nargs="*"`

```python
from flows.lib.shared_args import S3_PATH, default, multi

# Use multiple S3 paths
multi(S3_PATH)  # Allows: -s path1 path2 path3

# Set default for optional arg
default(WORK_DIR, "/tmp")  # Help text updates automatically
```

### **All Flows in `__main__` Block**

```python
if __name__ == "__main__":
    args = parse_args("Description of this flow.")
    my_flow(**vars(args))
```

## **6. Parsers — Plugin-Based Architecture**

Parsers transform raw input data into GenomeHubs-compatible TSV format.

### **Standard Signature**

```python
@flow()
def parse_<name>(working_yaml: str, work_dir: str, append: bool, **kwargs) -> None:
    """
    Parse <description>.

    Args:
        working_yaml (str): Path to the working YAML configuration file.
        work_dir (str): Path to the working directory for outputs.
        append (bool): If True, append to existing TSV file.
        **kwargs: Additional arguments passed through the pipeline.
    """
    config = load_config(working_yaml)
    # Parse logic here
    # Derive output path from config: config.meta['file_name']
    # Write to work_dir
```

### **Plugin Registration**

Every parser must define a `plugin()` function:

```python
def plugin():
    """Register the parser."""
    return Parser(
        name="NCBI_ASSEMBLIES",
        func=parse_ncbi_assemblies,
        description="Parse NCBI datasets JSONL into assemblies TSV.",
    )
```

### **Path Derivation**

Paths are derived from the YAML configuration to ensure consistency:

```python
config = load_config(working_yaml)
# Output filename defined in YAML: config.meta['file_name']
output_file = os.path.join(work_dir, config.meta['file_name'])
```

### **Standard Arguments**

All parsers inherit from `flows/parsers/args.py` and use:

- **INPUT_PATH** (`-i`): Raw input file
- **YAML_PATH** (`-y`): YAML configuration
- **APPEND** (`-a`): Flag to append vs. overwrite
- **DATA_FREEZE_PATH**: Optional frozen dataset list

## **7. Updaters — Event-Driven Data Fetching**

Updaters fetch remote data and write to local disk. They emit events for downstream automation.

### **Standard Pattern**

```python
@flow()
def update_<source>(root_taxid: str, output_path: str, s3_path: Optional[str] = None):
    """
    Fetch <description> and write to output file.

    Args:
        root_taxid (str): Root taxonomy ID for filtering.
        output_path (str): Local file path to write output.
        s3_path (str): Optional S3 path to compare against for change detection.
    """
    # Fetch data
    line_count = fetch_data(root_taxid, output_path)

    # Emit event for downstream flows
    emit_event(
        event="update.<source>.completed",
        resource={
            "prefect.resource.id": f"update.<source>.{root_taxid}",
            "prefect.resource.type": "update.<source>",
        },
        payload={"line_count": line_count},
    )
```

### **Event Emission**

Always emit completion events with:

- `event`: Short name for downstream filtering (`update.<name>.completed`)
- `resource.prefect.resource.id`: Unique identifier combining source and parameters
- `resource.prefect.resource.type`: Broad category for grouping
- `payload`: Data relevant to downstream flows (counts, status, etc.)

## **9. Orchestrators — Docker Isolation for Incompatible Dependencies**

### **When to Use Orchestrators**

⚠️ **RARE EDGE CASE ONLY.** Use orchestrators when:

1. An updater requires a package incompatible with Prefect/GenomeHubs
2. The conflict cannot be resolved (e.g., tol-sdk needs Pydantic < 2.0, Prefect needs 2.0+)
3. The updater works fine with `SKIP_PREFECT=true python3 -m flows.updaters.<updater>`

### **Why This Complexity?**

- **Standard case:** Updaters run in Prefect environment directly
- **Rare case:** Library version conflicts force Docker isolation
- **SOLUTION:** Orchestrator runs Prefect logic in a lightweight container, delegates heavy work to separate Docker container with `SKIP_PREFECT=true`

### **Architecture**

```
Prefect Deployment
    ↓
    Orchestrator Flow (@flow, in Prefect)
    ├─ Calls run_docker_flow() task
    │  └─ Runs: docker run -e SKIP_PREFECT=true python -m flows.updaters.<updater>
    │     (separate container with isolated dependencies)
    └─ Calls emit_completion_event() task
       └─ Triggers downstream flows
```

### **Pattern**

**Updater (unchanged, works with SKIP_PREFECT=true):**

```python
# flows/updaters/update_tol_genome_notes.py
@flow()
def update_tol_genome_notes(output_path: str, s3_path: str = None) -> None:
    # This will fail in Prefect (Pydantic conflict) but works with SKIP_PREFECT=true
    from tol.sources.portal import portal
    # ...
```

**Orchestrator (new wrapper):**

```python
# flows/orchestrators/tol_genome_notes_orchestration.py
from flows.orchestrators.tasks import run_docker_flow, emit_completion_event
from prefect import flow, get_run_logger

@flow(name="tol-genome-notes-orchestration")
def orchestrate_tol_genome_notes(
    output_path: str = "/tmp/tol_genome_notes.tsv",
    s3_path: str = None,
    min_records: int = 1,
) -> dict:
    """
    Run tol-genome-notes in Docker with dependency isolation.

    Prefect handles orchestration (lightweight).
    Docker handles tol-sdk execution (isolated environment).
    """
    logger = get_run_logger()

    result = run_docker_flow(
        flow_name="tol-genome-notes",
        flow_script="flows/updaters/update_tol_genome_notes.py",
        output_path=output_path,
        s3_path=s3_path,
        min_records=min_records,
    )

    emit_completion_event(
        flow_name="tol-genome-notes",
        result=result,
        resource_type="tol.genome.notes",
    )

    logger.info(f"🎉 Orchestration complete: {result}")
    return result
```

**Prefect deployment (register orchestrator, not raw updater):**

```yaml
deployments:
  - name: orchestrate-tol-genome-notes
    entrypoint: flows/orchestrators/tol_genome_notes_orchestration.py:orchestrate_tol_genome_notes
    schedules:
      - cron: "5 0 * * *"
```

### **Critical Constraints**

✅ **Orchestrators ARE for:** Updaters with incompatible dependencies

❌ **Orchestrators are NOT for:**

- Updaters that work fine in Prefect environment
- Parsers (they MUST work in Prefect, never introduce conflicts)
- Validators (they MUST work in Prefect, never introduce conflicts)

### **Dependency Rule for All Components**

| Component | Compatible with Prefect? | Use Orchestrator?   |
| --------- | ------------------------ | ------------------- |
| Parser    | ✅ MUST BE               | ❌ Never            |
| Updater   | ✅ Usually               | ⚠️ Only if conflict |
| Validator | ✅ MUST BE               | ❌ Never            |

**Parsers and Validators must NEVER introduce incompatible dependencies.**

## **9. Error Handling and Validation**

### **Input Validation — Early and Explicit**

Check inputs at task boundaries:

```python
def fetch_ncbi_datasets_sequences(accession: str) -> Generator[dict, None, None]:
    """Fetch sequence report from NCBI."""
    if not utils.is_safe_path(accession):
        raise ValueError(f"Unsafe accession: {accession}")

    if not accession.isdigit():
        raise ValueError(f"Invalid accession format: {accession}")

    # Proceed with safe inputs
```

### **Descriptive Exceptions**

Always raise with context:

```python
try:
    with open(jsonl_path, "r") as f:
        data = json.load(f)
except Exception as e:
    raise RuntimeError(f"Error reading JSONL file at {jsonl_path}: {e}") from e
```

### **Non-Critical Failures — contextlib.suppress**

For graceful degradation:

```python
import contextlib

# Load cached data, but don't fail if cache doesn't exist
with contextlib.suppress(Exception):
    cached = load_from_cache(cache_path)
    if cached:
        print(f"Using cached data from {cache_path}")
        return cached
```

### **Logging with print()**

Use `print()` for user-facing logs. Prefect captures stdout when `@task(log_prints=True)`:

```python
@task(log_prints=True)
def process_records(records: list):
    """Process records with logging."""
    print(f"Processing {len(records)} records...")
    for record in records:
        print(f"  Processed {record['id']}")
    print("Done.")
```

## **12. Adding New Flows**

### **Checklist**

- [ ] Use absolute imports: `from flows.lib import utils`
- [ ] Define CLI arguments in shared_args.py or subdir args.py
- [ ] Add comprehensive docstring (Google format)
- [ ] Use `@task()` and `@flow()` decorators via conditional_import
- [ ] Keep functions focused; extract if >3 nesting levels
- [ ] Emit events if an updater or async trigger needed
- [ ] Add `if __name__ == "__main__"` block for standalone execution
- [ ] Test with `SKIP_PREFECT=true python3 -m flows.<path>.<module>`
- [ ] If parser: define `plugin()` function and register in register.py

### **Parser Template**

```python

    parsed = {}

def plugin():
    """Register parser."""
```

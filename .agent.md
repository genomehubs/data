# GenomeHubs Data Flows — Agent Behavior Guide

This document guides AI agents working on the GenomeHubs data pipeline. Read `.copilot-instructions.md` for detailed conventions first.

## **Agent Directives**

### **1. When Creating New Flows**

**Always verify before implementing:**

```
1. ✓ Is this a parser (transforms input data)?
   → Create in flows/parsers/ with plugin registration

2. ✓ Is this an updater (fetches remote data)?
   → Create in flows/updaters/ with event emission

3. ✓ Is this a validator (checks data quality)?
   → Create in flows/validators/ with planned subdirectory structure
   → For now: validate_file_pair.py as reference in flows/lib/

4. ✓ Is this shared infrastructure?
   → Add to flows/lib/utils.py or flows/lib/shared_*.py
```

### **2. Import Rule — NO EXCEPTIONS**

Every internal import must be absolute package imports:

```python
# ✓ ALWAYS USE THIS:
from flows.lib import utils
from flows.lib.conditional_import import flow, task
from flows.lib.shared_args import parse_args, required

# ✗ NEVER:
import utils
from conditional_import import ...
from shared_args import ...
```

**Agent validation:** After writing any file, grep for bare imports of lib modules. Reject and fix if found.

### **3. Docstring Requirement**

Every public function must have a docstring in Google format:

```python
def my_function(arg1: str, arg2: int) -> bool:
    """
    Brief description of what this does.

    Optional extended description if needed.

    Args:
        arg1 (str): Description of arg1.
        arg2 (int): Description of arg2.

    Returns:
        bool: Description of return value.

    Raises:
        ValueError: When arg2 is negative.
    """
```

**Agent validation:** After writing any function, verify docstring exists and matches Google format.

### **4. Function Complexity Rules**

- ✓ Nesting **≤ 3 levels** (if/for/while chains)
- ✓ Single responsibility per function
- ✓ Minimize side effects in utility functions
- ✓ Use generators for memory-intensive loops

**Agent validation:** If a function has >3 nesting levels, extract inner logic into a separate helper function.

### **5. CLI Arguments — Use Dict-Based Shared Args**

New arguments:

1. Define in `flows/lib/shared_args.py` if used across directories
2. Reference in subdir `args.py`:

```python
# flows/parsers/args.py
from flows.lib.shared_args import INPUT_PATH, YAML_PATH, APPEND, required, parse_args

def parse_args(description="A parser"):
    return _parse_args([required(INPUT_PATH), required(YAML_PATH), APPEND], description)
```

**Agent validation:** No CLI argument parsing in individual flows. All use `parse_args()` wrapping shared dicts.

### **6. Parser Pattern — Plugin Registration Required**

Every parser must follow this structure:

```python
@flow()
def parse_my_data(working_yaml: str, work_dir: str, append: bool, **kwargs) -> None:
    """Parse my data format."""
    config = load_config(working_yaml)
    # ...

def plugin():
    """Register parser."""
    return Parser(
        name="MY_NAME",
        func=parse_my_data,
        description="Parse my data.",
    )

if __name__ == "__main__":
    args = parse_args("Parse my data.")
    parse_my_data(**vars(args))
```

**Agent validation:** Verify `plugin()` function exists and matches structure above.

### **7. Updater Pattern — Emit Events**

Every updater must emit completion events:

```python
@flow()
def update_my_source(root_taxid: str, output_path: str):
    """Fetch data and emit completion event."""
    line_count = fetch_and_write(output_path)

    emit_event(
        event="update.my_source.completed",
        resource={
            "prefect.resource.id": f"update.my_source.{root_taxid}",
            "prefect.resource.type": "update.my_source",
        },
        payload={"line_count": line_count, "status": "success"},
    )
```

**Agent validation:** Check that every updater has an `emit_event()` call.

### **7a. Orchestrators — ONLY When Dependencies Conflict**

**⚠️ Rare edge case.** ONLY create an orchestrator if:

- The updater requires a package incompatible with Prefect (e.g., Pydantic version conflict)
- The updater cannot be modified to avoid the conflict
- The updater works fine with `SKIP_PREFECT=true`

**Example:** tol-sdk needs Pydantic < 2.0, but Prefect 3.x needs Pydantic 2.0+

**Pattern:**

```python
# flows/orchestrators/tol_genome_notes_orchestration.py
from flows.orchestrators.tasks import run_docker_flow, emit_completion_event
from prefect import flow, get_run_logger

@flow(name="tol-genome-notes-orchestration")
def orchestrate_tol_genome_notes(output_path: str, s3_path: str = None, min_records: int = 1) -> dict:
    """Run tol-genome-notes in Docker to isolate Pydantic conflicts."""
    result = run_docker_flow(
        flow_name="tol-genome-notes",
        flow_script="flows/updaters/update_tol_genome_notes.py",
        output_path=output_path,
        s3_path=s3_path,
        min_records=min_records,
    )
    emit_completion_event(flow_name="tol-genome-notes", result=result, resource_type="tol.genome.notes")
    return result
```

**Critical constraint:** Orchestrators are **ONLY for updaters**. Parsers and validators MUST NEVER introduce incompatible dependencies.

**Agent validation:**

- If updater raises Pydantic/dependency error, check if orchestrator already exists
- Never create orchestrator unless dependency conflict is confirmed
- Verify orchestrator is registered in `flows/prefect.yaml` (not raw updater)

### **8. Validator Pattern — Forward to New Structure**

**Current state:** Validation in `flows/lib/validate_file_pair.py`

**Planned state:** Move to `flows/validators/` directory

**Agent behavior:**

```python
    min_assigned: int = 1

    """Validate data against schema."""
    # ...
    return validation_passed
```

- Use standard validator args from planned `flows/validators/args.py`

## **Common Tasks**

### **Task: Add a New Shared Util Function**

**Steps:**

1. Add function to `flows/lib/utils.py`
2. Add comprehensive docstring (Google format)
3. Keep function pure (read-only or single-write)
4. Test: `SKIP_PREFECT=true python3 -c "from flows.lib.utils import my_func; my_func(...)"`

**Example:**

```python
def extract_accession_version(accession: str) -> tuple:
    """
    Extract version number from NCBI accession.

    Args:
        accession (str): NCBI accession, e.g., "GCA_000001405.30".

    Returns:
        tuple: (base_accession, version_number).

    Raises:
        ValueError: If accession format is invalid.
    """
    if not accession or '.' not in accession:
        raise ValueError(f"Invalid accession format: {accession}")
    base, version = accession.rsplit('.', 1)
    return (base, int(version))
```

### **Task: Add a New CLI Argument**

**Steps:**

1. Add dict to `flows/lib/shared_args.py`
2. Use in subdir `args.py` with `required()` / `default()` / `multi()`
3. Test: `python3 -m flows.lib.validate_file_pair --new_arg -h`

**Example:**

```python
# flows/lib/shared_args.py
RETRY_COUNT = {
    "flags": ["--retry_count"],
    "keys": {
        "help": "Number of retries on failure.",
        "default": 3,
        "type": int,
    },
}

# Usage in updater
from flows.lib.shared_args import RETRY_COUNT, default

args_list = [
    required(OUTPUT_PATH),
    default(RETRY_COUNT, 5),  # Override default
]
```

### **Task: Add a New Parser**

**Steps:**

1. Create `flows/parsers/parse_<name>.py`
2. Implement `parse_<name>(working_yaml, work_dir, append, **kwargs)`
3. Add `plugin()` function returning `Parser(...)`
4. Add docstring and type hints
5. Test: `SKIP_PREFECT=true python3 -m flows.parsers.parse_<name> -h`

**Checklist:**

- [ ] Inherits from `flows/parsers/args.py` (INPUT_PATH, YAML_PATH, APPEND, DATA_FREEZE_PATH)
- [ ] Docstring describes transformation
- [ ] Paths derived from config: `config.meta['file_name']`
- [ ] Output written to `work_dir`
- [ ] `plugin()` function present and correctly structured

### **Task: Add a New Updater**

**Steps:**

1. Create `flows/updaters/update_<name>.py`
2. Implement `update_<name>(root_taxid: str, output_path: str, ...)`
3. Emit completion event with `emit_event()`
4. Add docstring and type hints
5. Test: `SKIP_PREFECT=true python3 -m flows.updaters.update_<name> -h`

**Checklist:**

- [ ] Uses standard updater args: ROOT_TAXID, OUTPUT_PATH, S3_PATH, etc.
- [ ] Docstring explains what is fetched
- [ ] Data written to `output_path`
- [ ] Event emitted on completion with appropriate `payload`
- [ ] Error handling with descriptive exceptions

### **Task: Refactor Flow to Use Shared Utils**

**Steps:**

1. Identify repeated logic in multiple flows
2. Extract to function in `flows/lib/utils.py`
3. Update all flows to import and use the new utility
4. Verify with local tests: `SKIP_PREFECT=true python3 -m flows.<path>.<flow> -h`

**Example:**
If 3 parsers all validate accessions the same way:

```python
# flows/lib/utils.py
def validate_ncbi_accession(accession: str) -> bool:
    """Check if accession is valid NCBI format."""
    return bool(re.match(r"^[GG][C]F_\d{9}\.\d+", accession))

# In each parser
from flows.lib.utils import validate_ncbi_accession

if not validate_ncbi_accession(accession):
    print(f"Skipping invalid accession: {accession}")
    continue
```

## **Code Review Checklist — For Agents to Self-Check**

Before implementing, verify:

- [ ] **Imports**: All internal imports use `flows.lib.*` absolute paths
- [ ] **Docstrings**: All public functions have Google-format docstrings
- [ ] **Type hints**: All function signatures include type annotations
- [ ] **Nesting**: No function has >3 levels of control flow nesting
- [ ] **CLI args**: All arguments use dict-based pattern from shared_args.py
- [ ] **Logging**: Uses `print()` for output; tasks have `@task(log_prints=True)`
- [ ] **Error handling**: Raises descriptive exceptions, validates inputs early
- [ ] **Side effects**: Utility functions are pure or single-write
- [ ] **Testing**: Tested with `SKIP_PREFECT=true python3 -m flows.<path>`
- [ ] **Pattern compliance**:
  - Parser? Has `plugin()` function and plugin registration
  - Updater? Emits completion event
  - Shared util? Added to `flows/lib/utils.py` with docstring
  - Validator? Planned for `flows/validators/` directory

## **Testing Guidelines for Agents**

### **Local Smoke Test — Always Run**

```bash
# Before committing, verify with SKIP_PREFECT=true
SKIP_PREFECT=true python3 -m flows.<path>.<module> -h

# Should print usage without ModuleNotFoundError
```

### **Functional Test — After Implementation**

```bash
# Example parser
SKIP_PREFECT=true python3 -m flows.parsers.parse_my_data \
  -i /path/to/input.jsonl \
  -y /path/to/config.yaml

# Example updater
SKIP_PREFECT=true python3 -m flows.updaters.update_my_source \
  -r 2759 \
  -o /tmp/output.jsonl

# Example validator
SKIP_PREFECT=true python3 -m flows.validate_file_pair \
  -y /path/to/config.yaml \
  -w /tmp
```

### **Type Checking — Optional but Recommended**

```bash
python3 -m mypy flows/lib/utils.py --ignore-missing-imports
```

## **Special Cases and Exceptions**

### **When to Deviate from Dict-Based Args**

❌ **Never**. Always use dict-based args in `shared_args.py`.

### **When to Add Side Effects to Utility Functions**

❌ **Avoid**. If needed, document clearly:

```python
def cache_result(key: str, value: Any) -> None:
    """
    Store result in global cache (side effect).

    Note: This function has a side effect — it modifies module-level state.
    Use sparingly for performance-critical operations.
    """
    _CACHE[key] = value
```

### **When to Nest Beyond 3 Levels**

❌ **Extract to separate function**:

```python
# Bad: 4 levels
for a in items:
    if a.valid:
        for b in a.subitems:
            if b.active:
                for c in b.children:
                    if c.required:
                        process(c)

# Good: Extract inner loops
def process_children(b):
    for c in b.children:
        if c.required:
            process(c)

def process_subitems(a):
    for b in a.subitems:
        if b.active:
            process_children(b)

for a in items:
    if a.valid:
        process_subitems(a)
```

## **When to Escalate to Humans**

- 🚨 Changes to Prefect deployment configuration (`prefect.yaml`)
- 🚨 Changes to the fetch-parse-validate orchestration logic
- 🚨 Breaking changes to shared CLI argument structure
- 🚨 Uncertainty about parser/updater/validator categorization
- 🚨 Performance implications requiring profiling

For these, always request human review before proceeding.
